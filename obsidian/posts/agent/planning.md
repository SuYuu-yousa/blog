Planning的不同，本质上就是怎么去组织不同任务步骤发给llm的结构/顺序



## Tree of Thoughts (ToT)

之前那几个都是**一条线**往下走，ToT 是**展开多条路**，选最好的：

```
                    问题
                   / | \
                思路A 思路B 思路C
               /  \    |    \
             A1   A2   B1   C1
             ✗    ✓    ✓     ✗
                  |    |
                 A2a  B1a
                  ✓    ✗
                  |
                最终答案
```

本质上就是：

```
第一轮：调LLM 3次，生成3种不同思路
       ↓
       调LLM("请评估这3��思路哪个更靠谱")
       ↓ 淘汰掉差的
       
第二轮：对留下的思路，各自再往下展开
       ↓
       调LLM("再评估一轮")
       ↓ 继续淘汰
       
...直到找到最优路径
```

**特点：花更多 LLM 调用次数来换质量，适合数学/推理/复杂决策类问题**

**代价也很明显：���。调用次数指数级增长。**

## 所有主流 Planning 方法一览





```
简单 ──────────────────────────────────── 复杂
 │                                         │
 │  直接问  CoT   ReAct  Plan&Exe  ToT  多Agent协作
 │   │       │      │       │       │      │
 │  1次    1次    多次循环  1+N次  很多次  多个Agent
 │  调用    调用    调用     调用    调用   各自循环
```

### 1. 直接调用（最简单，严格说不算 planning）

```
调LLM("回答这个问题") → 答案
```

就一次，不多想

### 2. CoT（Chain of Thought）

```
调LLM("请一步步想再回答") → 带推理过程的答案
```

还是一次调用，只是 prompt 里要求它想细点

### 3. Self-Consistency（CoT 的加强版）

```
调LLM 同一个问题 5 次（用 CoT）
→ 得到 5 个答案
→ 投票，选出现最多的那个
```

**用"多数投票"来提高准确率，简单粗暴但有效**

### 4. ReAct

```
Think → Act → Observe → Think → Act → Observe → ... → 答案
```

边想边做，一步一步循环

### 5. Plan-and-Execute

```
先出完整计划 → 逐步执行 → 答案
```

先想完再做

### 6. Reflexion（反思机制）🔥比较火

```
第一轮：正常做任务 → 得到结果
        ↓
调LLM("你做得对吗？哪里可以改进？")
        ↓ 返回：反思/批评
第二轮：带着反思重新做 → 得到更好的结果
        ↓
调LLM("这次做得对吗？")
        ↓
...直到满意
```

**本质：让 LLM 自己当自己的 code reviewer，做完了回头审视**

### 7. LATS（Language Agent Tree Search）🔥较新

```
ToT + ReAct + 蒙特卡洛树搜索

不只展开多条思路（ToT）
每条思路还能调工具与环境交互（ReAct）
用搜索算法选最优路径（MCTS）
```

**可以理解为 ToT 的加强版，更适合需要与外部交互的复杂任务**

### 8. Multi-Agent Debate（多Agent辩论）🔥比较火

```
AgentA: 我觉得答案是 X，因为...
AgentB: 我不同意，应该是 Y，因为...
AgentC: 综合来看 A 说的更有道理，但需要修改...

→ 几轮辩论后收敛到最终答案
```

**本质：同一个LLM扮演不同角色，互相质疑，提高质量**

## 汇总对比

| 方法                 | 调LLM次数 | 核心思想         | 适用场景       |
| -------------------- | --------- | ---------------- | -------------- |
| **直接调用**         | 1次       | 直接问           | 简单问题       |
| **CoT**              | 1次       | 想细点           | 推理题         |
| **Self-Consistency** | N次       | 多做几次投票     | 要求准确       |
| **ReAct**            | 循环多次  | 边想边做边看     | 需要调工具     |
| **Plan-Execute**     | 1+N次     | 先规划再做       | 多步骤复杂任务 |
| **Reflexion**        | 至少2轮   | 做完回头反思再做 | 代码生成/写作  |
| **ToT**              | 很多次    | 展开多条路选最优 | 数学/逻辑难题  |
| **LATS**             | 非常多    | ToT+ReAct+搜索   | 复杂交互任务   |
| **Multi-Agent**      | 非常多    | 多角色辩论/协作  | 需要多视角     |

## 回到本质

你之前总结得很对，我再帮你加一层：

> **所有这些方法的区别，就是三个变量的排列组合：**
>
> 1. **调几次 LLM**（1次 / 多次循环 / 多条并行）
> 2. **每次给 LLM 什么上下文**（之前结果 / 反思 / 其他Agent的意见）
> 3. **怎么从多个结果里选最终答案**（直接用 / 投票 / 评分 / 搜索）

万变不离其宗，底下都是在反复调那个文本进文本出的接口。
