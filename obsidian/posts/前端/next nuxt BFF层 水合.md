# 基本概念-CSR SSR SSG

- **Next.js** 是基于 **React** 的框架

- **Nuxt.js** 是基于 **Vue** 的框架

- 它们常被叫做 **全栈框架**

- 它们支持 **SSR / SSG / CSR** 等页面渲染方式

- **确实会让 React/Vue 的一部分代码在服务器运行**

- 但不是简单地“把整个前端搬到服务器”，而是：

  - 有些代码在服务器跑

  - 有些代码在浏览器跑

  - 两边协作完成网页


------

## 什么叫“全栈框架”？

> **全栈框架 = 不只管前端页面，还顺带帮你处理服务端能力的框架**

普通 React / Vue 更多像是“写界面”的工具。
而 Next / Nuxt 不只是写界面，还会帮你处理：

- 路由 页面渲染  服务端渲染
- 静态生成  接口/API  部署优化
- SEO 相关能力

所以它不是单纯前端框架，而是“前端 + 一部分后端能力”都给你打包好了。

------

| 方式    | 一句话总结                       |
| ------- | -------------------------------- |
| **SSR** | 页面是“访问时在服务器现做的”     |
| **SSG** | 页面是“打包时提前做好的”         |
| **CSR** | 页面是“浏览器自己用 JS 画出来的” |

| 方式    | 全称                   | 渲染时机         | 页面内容怎么来                                 | 优点                                   | 缺点                                                         | 适合场景                                                     |
| ------- | ---------------------- | ---------------- | ---------------------------------------------- | -------------------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **SSR** | Server-Side Rendering  | **请求时**       | 服务器先生成完整 HTML，再返回给浏览器          | 首屏快、SEO 好、适合动态内容           | 服务器压力更大；每次请求都可能重新生成；要注意服务端/浏览器环境差异 | 内容经常变化，且需要 SEO。新闻、电商详情、搜索页、官网首页   |
| **SSG** | Static Site Generation | **构建时**       | build 时提前生成 HTML，访问时直接返回静态文件  | 速度快、SEO 好、服务器压力小、部署简单 | 内容更新后通常要重新构建；不适合高频变化页面                 | 内容更新不频繁，可提前生成。博客、文档、公司官网、落地页     |
| **CSR** | Client-Side Rendering  | **浏览器运行时** | 先返回基础 HTML，再下载 JS，由前端在浏览器渲染 | 前后端分离清晰、交互强、适合应用型系统 | 首屏可能慢、SEO 相对弱                                       | 更偏应用型，SEO 不重要，交互复杂。后台管理、数据看板、用户中心、内部系统 |



## Next / Nuxt 是什么？

你可以理解成：

> 它们给 React/Vue 提供了“多种渲染方式 + 服务端能力 + 工程化能力”

它们并不是只做 SSR，实际上它们帮你处理了：

- **页面路由**
- **服务端渲染 SSR**
- **静态生成 SSG**
- **客户端渲染 CSR**
- **接口/API**
- **打包构建**
- **性能优化**
- **SEO 支持**

所以你开发网站时，不用从零拼各种工具。

------

## “让 React 和 Vue 在服务器上运行”这句话对吗？

**对，但不完整。**

更准确说法是：

> Next / Nuxt 可以让 React / Vue 组件在服务器端先执行一次，生成 HTML，再把结果发给浏览器。

注意重点不是“永远在服务器跑”，而是：

- **首轮渲染** 可以在服务器
- **后续交互** 主要还是在浏览器

你访问一个商品页：

**使用 Next SSR 时**

1. 浏览器请求 `/product/123`
2. Next 服务器拿到请求
3. 去数据库/接口取商品数据
4. 用 React 组件生成 HTML
5. 把 HTML 发给浏览器
6. 浏览器先显示商品信息
7. JS 加载后，按钮点击、加入购物车等交互开始生效

所以是：

- **HTML 先由服务器生成**
- **交互后由浏览器接管**

------

## 同构 / 通用”思想？

你以后可能会听到两个词：

- **同构**
- **isomorphic**
- **universal rendering**

大意是：

> 同一套组件代码，既能在服务器运行，也能在浏览器运行

比如同一个 React 组件：

```
export default function Page() {
  return <h1>Hello</h1>
}
```

这段组件代码：

- 可以在服务器执行，生成 `<h1>Hello</h1>`
- 也可以在浏览器执行，绑定事件、完成交互

这就是 Next/Nuxt 很核心的思想。

## Next怎么做的？

基本没有办法能强制next走SSR还是SSG，一个项目的不同页面，一个页面的不同组件，可能走的策略都是不一样的，那么next怎么自行判断的呢？

####  Next 的内部判断粗略流程

很粗略地说，它像这样判断：

1. 这个路由有没有明显的动态依赖？
   - `cookies()`
   - `headers()`
   - `no-store`
   - 某些请求相关数据
     有的话 → 动态 SSR
2. 如果没有动态依赖，数据能缓存吗？
   - 能 → 静态生成 / ISR
3. 页面里有没有客户端组件？
   - 有 → 这些组件的交互逻辑打包成客户端 JS
   - 没有 → 页面可以更偏服务端输出

所以最后一个页面可能同时包含：

- 服务端生成的 HTML

- 给客户端组件用的 JS

- 一些 hydration 信息

  

  

  

  

  

  

  

# ==Next.js?==

可以把 Next.js 理解成“套在 React 外面的一层工程化运行时 + 约定式框架”。你不是把现有 React 项目原封不动包一下，它就自动全优化了；而是你要**按 Next 的方式组织项目**，它才能接管路由、服务端渲染、静态生成、打包优化、图片优化、接口路由这些能力。也就是说，**Next 会帮你做很多优化，但前提是你得按它的规则写**。比如你还是把所有页面都写成 `'use client'`，所有数据都在 `useEffect` 里请求，那它也帮不了你太多，本质上还是 CSR。真正有收益的是：你知道什么时候让页面保持服务端组件、什么时候缓存、什么时候动态、什么时候拆客户端组件。
你如果说“我会 Next.js”，面试官通常不会只停留在“Next 是 React 框架”这层，往下大概率会问这几类东西：

| 方向                    | 常问点                                                       |
| ----------------------- | ------------------------------------------------------------ |
| 基础定位                | Next.js 和 React 什么关系；为什么要用 Next；解决了 React 哪些问题 |
| 渲染模式                | SSR / SSG / ISR / CSR 区别；App Router 下怎么实现；什么时候选哪种 |
| Server/Client Component | 什么是服务端组件；为什么默认是服务端组件；`'use client'` 的作用；二者边界怎么划分 |
| 数据获取                | 在 App Router 里怎么拉数据；`fetch` 缓存策略；`revalidate`；`no-store`；为什么不用老的 `getServerSideProps` |
| 路由                    | app 目录路由、动态路由、嵌套路由、layout、loading、error、not-found |
| 性能优化                | 代码分割、流式渲染、图片优化、字体优化、预取、缓存、减少客户端 JS |
| 工程能力                | 中间件 middleware、API Route / Route Handler、鉴权怎么做、部署到 Vercel 或 Node 环境 |
| 实战理解                | 做过什么页面；某个页面为什么用 SSR/SSG；怎么处理登录态；怎么处理 SEO |

如果你是校招生，别人最爱问的不是很偏门的源码，而是“你到底真的用过没”。所以你最好把下面这些讲顺：1）为什么 Next 比纯 React 更适合内容型网站；2）新版 App Router 里默认是服务端组件；3）客户端组件会增加 JS 体积，所以不要乱加 `'use client'`；4）怎么通过 `fetch` 的缓存策略让页面静态或动态；5）layout、page、loading、error 分别干嘛；6）登录态页面怎么处理。
你可以把“会 Next.js”拆成一套面试可说的话：**我会用 Next.js App Router 做基于 React 的全栈应用，知道怎么按页面选择 SSG/SSR/CSR，知道服务端组件和客户端组件的边界，知道用 `fetch` 缓存、`revalidate`、`no-store` 控制渲染策略，也会用 layout、动态路由、route handler、middleware 来做页面组织、接口和鉴权。** 这句话就比较像真的做过。
再具体一点，面试官很可能追问这些问题，你最好会答：

| 问题                                       | 你至少要会说的点                                             |
| ------------------------------------------ | ------------------------------------------------------------ |
| Next 和 React 的区别？                     | React 是 UI 库；Next 是基于 React 的全栈框架，提供路由、SSR/SSG、构建优化、接口能力等 |
| App Router 和以前 Pages Router 区别？      | App Router 支持 Server Component、嵌套路由、layout、loading、streaming，数据获取方式也变了 |
| 为什么默认是 Server Component？            | 减少下发给浏览器的 JS，提升首屏和性能，适合把数据获取放服务端 |
| 什么情况下要加 `'use client'`？            | 用状态、事件、生命周期、浏览器 API、第三方客户端库时         |
| `'use client'` 有什么代价？                | 组件及其依赖会进客户端包，增加 JS 体积                       |
| 怎么做 SSR / SSG？                         | 在 App Router 里主要靠 `fetch` 缓存策略和 `revalidate`，而不是老的 `getServerSideProps/getStaticProps` |
| `cache: 'no-store'` 和 `revalidate` 区别？ | `no-store` 每次请求都重新取，偏 SSR；`revalidate` 是静态基础上定时更新，偏 ISR |
| 为什么 Next 页面还能写 `async` 组件？      | 因为服务端组件可以直接 await 数据                            |
| layout 有什么用？                          | 复用页面壳、导航、侧边栏，嵌套路由共享布局且切页时不一定整页重渲染 |
| loading.tsx / error.tsx 是什么？           | 路由级加载态和错误边界                                       |
| 怎么做鉴权？                               | 可以 middleware 做拦截；服务端读 cookie/session；客户端只负责展示交互 |
| Next 怎么做接口？                          | `app/api/.../route.ts` 写 Route Handler                      |
| SEO 怎么做？                               | 用服务端渲染/静态生成 + `generateMetadata` 配置标题、描述等  |

你还得有一点“避坑认知”，这也很像真正做过项目的人会说的话：第一，不要一上来全写客户端组件，不然就退化成普通 React SPA 了；第二，服务端组件里不能直接用 `useState/useEffect` 和浏览器 API；第三，服务端组件获取的数据可以直接参与首屏 HTML 生成，这对 SEO 和首屏很有帮助；第四，客户端组件适合放交互重的部分，比如评论、弹窗、图表、播放器、表单复杂逻辑；第五，页面策略是可以混用的，不是全站只能一种。
如果你问“那我到底需要加点啥”，答案是：你需要学会按 Next 的目录和约定来写，至少包括 `app/` 路由、`layout.tsx`、`page.tsx`、服务端取数、客户端组件、路由跳转、API route、基础部署。不是给 React 项目装个依赖就结束了，而是开发方式要切到 Next 的范式。
如果你是校招生，我建议把准备重点放在“能解释清楚 + 能举项目例子”，而不是卷太深的源码。你至少要能讲一个完整案例：比如“我做了一个博客/商城项目，首页和文章页用了静态生成，商品详情用了动态渲染，评论区和购物车用了客户端组件，登录态通过 cookie + middleware 控制，接口通过 route handler 提供”。这比背很多术语有用。
如果你愿意，我下一条可以直接给你一份 **“校招面试里 Next.js 高频问题 + 标准短答案”**，你可以直接背。
