# 浏览器进程

浏览器通常至少包含浏览器主进程、渲染进程、网络相关进程或服务、GPU 进程，另外还可能有插件进程、扩展进程、音视频相关进程、存储和 utility 进程

==主进程就是管多少个标签页之类的==

==渲染进程一般每个页面一个，画这个页面==

==网络进程字面意思==

==GPU进程字面意思==

浏览器主进程负责整体协调，像标签页管理、地址栏、前进后退、窗口界面、部分权限控制、进程调度这些都归它管。

渲染进程负责某个页面的渲染工作，也就是解析 HTML/CSS、构建 DOM、样式计算、布局、绘制，以及执行页面里的 JavaScript。

GPU 进程负责图形相关加速，比如合成、光栅化等。

网络部分在现代浏览器里往往被单独抽成网络服务。



# 页面怎么来？

你现在输入了一个url

### 1.从url输入 到 准备请求

浏览器拿到 URL 后，先判断输入内容是不是一个可导航的地址(不是可能会转给搜索引擎)，再把它解析成规范化的 URL 结构，然后判断这次变化是不是一次真正的页面导航，比如只是 hash 变化就未必会重新请求；如果需要导航，它会优先检查本地状态和可复用资源，比如缓存、已有连接、已有网络信息，确认之后才进入真正的网络获取流程，也就是后面的 DNS、建连、发请求。你可以先把最开始记成一句话：==识别输入、规范 URL、判断导航、尝试复用、准备请求。==

###### q1)浏览器怎么判断你输入的是 URL 还是搜索词

浏览器先尝试按 URL 语法解析，能构成合法 URL 就直接导航，否则当搜索关键字处理。

###### q2）URL 怎么拆分，各部分是什么，标准长相

标准结构可以抽象成这一行：
`scheme://userinfo@host:port/path?query#fragment`
用一个例子：`https://user:pass@www.example.com:8080/path/to/page?name=alice&age=20#section1`，对应关系是：`https` 是协议（scheme），`user:pass` 是用户信息（一般现在很少用），`www.example.com` 是主机（host），`8080` 是端口（port，省略时 http 默认 80、https 默认 443），`/path/to/page` 是路径（path），`?` 后面 `name=alice&age=20` 是查询参数（query，服务器会收到），`#section1` 是片段标识（fragment，只在浏览器本地用，**不会**发给服务器）。浏览器内部会把这个字符串解析成一个结构化对象（类似你在 JS 里 new URL 得到的那种），后面所有流程都基于这个拆好的结构，不再拿原始字符串乱切。

### 2.发请求到返回入口文件

当浏览器已经确定“这是一个合法 URL，要导航过去”，下一步首先是做标准化和分类：比如把省略的协议补成 `https://`，把大小写无关的部分统一成规范形式，区分这次是不是只改了 hash（比如 `/#a` → `/#b`，这类叫 same-document navigation，通常不重新发请求），以及判断是当前标签页内导航还是要开新标签、新窗口。确认为“需要真正加载新文档”后，浏览器会选择或创建合适的渲染进程，并在发起网络请求之前，先尝试利用本地已有的信息：检查是否有可以直接命中的缓存、是否存在与该站点可复用的连接、是否有对应的 Service Worker 可以拦截请求；如果这些都不满足，才进入外部网络阶段，也就是基于 URL 中的 host 做 DNS 解析，基于 scheme 与端口建立或复用 TCP 连接、在 HTTPS 下完成 TLS 握手，然后构造 HTTP 请求报文发给服务器。下一步就是服务器返回响应、浏览器拿到主文档后启动“解析 + 渲染”流程，这块我们可以单独展开讲一轮。









# 强缓存和协商缓存

流程很简单：

1. 浏览器发请求
2. 服务器返回 HTML
3. 返回内容里分两部分理解：
   - **响应头**：先告诉浏览器“这页面该怎么处理”
   - **响应体**：真正的 HTML 内容

比如服务器会先说：

- 这个页面是什么编码
- 能不能执行外部脚本
- 能不能加载第三方资源
- 要不要缓存

然后浏览器再去解析 HTML。
